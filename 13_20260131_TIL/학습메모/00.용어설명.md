
## JSON(JavaScript Object Notation)
- 경량 데이터 교환 형식
- 사람이 읽고 쓰기 쉽고, 기계가 분석하고 생성하기 쉬움
- 주로 웹 애플리케이션에서 서버와 클라이언트 간 데이터 교환에 사용
- 예시:
 ```JSON
 {
   "name": "John Doe",
   "age": 30,
   "isStudent": false,
   "courses": ["Math", "Science", "History"],
   "address": {
     "street": "123 Main St",
     "city": "Anytown",
     "zip": "12345"
   }
 }
 ```

## XML(eXtensible Markup Language)
- 특정 목적에 맞게 데이터를 구조화하고 저장하기 위한 마크업 언어
 마크업이란? 데이터를 설명하는 태그를 추가하는 것
- 데이터를 계층 구조로 표현할 수 있으며, 사용자 정의 태그를 사용하여 다양한 형식의 데이터를 표현 가능
- 예시:
 ```XML
 <person>
  <name>John</name>
  <age>30</age>
  <city>New York</city>
</person>
```

## 아키텍쳐(Architecture)
- 뜻: **전체적인 건물 설계도와 운영 수칙**
- 소프트웨어 아키텍쳐: 프로그램이 단순히 돌아가게 하는것이 아니라, 추후 기능을 추가하고 에러를 고치기 쉽게 구조적으로 나누는 것이 핵심.
- 가장 대중적인 아키텍쳐: @Controller, @Service, @Repository 3가지 컴포넌트로 나누는 3계층 아키텍쳐
- 각 컴포넌트 역할
 1. @Controller: 사용자의 요청을 받고, 처리 결과를 반환하는 역할
 2. @Service: 비즈니스 로직을 처리하는 역할
 3. @Repository: 데이터베이스와 직접 소통하는 역할

## API(Application Programming Interface)
쉬운표현: API는**서로 다른 프로그램들**이 맞물려 돌아가게 해주는 연결고리
Application Programming 뜻: 응용 프로그래밍
Interface 뜻: 인터페이스(접점)
**모아서: 응용 프로그램들이 서로 소통할 수 있게 해주는 접점**
- 뜻: 서로 다른 소프트웨어 프로그램들이 **서로 소통**하고 **데이터를 주고받을**수 있게 해주는 매게체이자 약속
- 예시: 구글 맵 API, 트위터 API, 페이스북 API 등
  1. 구글 맵 API: 개발자가 자신의 애플리케이션에 구글 지도를 삽입하고, 위치 기반 서비스를 제공할 수 있게 해줌
   - 소통대상1: 구글지도, 소통대상2: 개발자 애플리케이션, 소통방법: API
  2. 트위터 API: 개발자가 트위터 데이터를 읽고 쓰거나, 트윗을 게시하고 사용자 정보를 조회할 수 있게 해줌
   - 소통대상1: 트위터, 소통대상2: 개발자 애플리케이션, 소통방법: API
  3. 페이스북 API: 개발자가 페이스북의 소셜 그래프 데이터를 활용하여 사용자 인증, 친구 목록 조회, 게시물 작성 등을 할 수 있게 해줌
  - 소통대상1: 페이스북, 소통대상2: 개발자 애플리케이션, 소통방법: API
  4. 날씨 API: 개발자가 실시간 날씨 정보를 자신의 애플리케이션에 통합할 수 있게 해줌
  - 소통대상A: 날씨 서비스, 소통대상B: 개발자 애플리케이션, 소통방법: API 
  5. 결제 게이트웨이 API: 개발자가 신용카드 결제, 전자지갑 결제 등 다양한 결제 수단을 애플리케이션에 통합할 수 있게 해줌
  - 소통대상 A: 결제 게이트웨이, 소통대상 B: 개발자 애플리케이션, 소통방법: API
  6. 유튜브 API: 개발자가 유튜브 동영상을 검색, 재생, 업로드할 수 있게 해줌
  - 소통대상 A: 유튜브, 소통대상 B: 개발자 애플리케이션, 소통방법: API
  7. 페이팔 API: 개발자가 페이팔 결제 시스템을 자신의 애플리케이션에 통합할 수 있게 해줌
  - 소통대상 A: 페이팔, 소통대상 B: 개발자 애플리케이션, 소통방법: API

## REST(Representational State Transfer)
  - Representational: 표현하는, State: 상태를 나타내는, Transfer: 전송하는
   자원을 이름으로 구분하여 해당 자원의 상태(정보)를 주고받는 모든 것
  1. Representational(표현): 자원(데이터)를 어떻게 보여줄지 결정하는 것(요즘은 주로 JSON 형식 사용)
  2. State(상태): 데이터의 현재 모습
  3. Transfer(전송): 그 데이터를 주고받는다는 뜻
   요약: 자원(데이터)을 특정한 형식으로 표현하여, 그 상태를 주고받는 방식
   예시: 블로그 게시물, 사용자 정보, 상품 목록 등
  - /product/1(1번 상품): 사용자가 /product/1로 GET 요청을 보내면, 서버는 1번 상품의 현재 상태(정보)를 JSON 형식으로 표현하여 응답
  - HTTP메서드로 무엇을 할지 정함
    1. GET: 조회, POST: 생성, PUT: 수정, DELETE: 삭제
  4. 표현(Representation): 자원의 상태를 클라이언트에게 전달하는 형식
    - 예: JSON, XML 등

## RESTful and NON-RESTful
- 뜻: REST 아키텍처의 원칙을 성실하게 지킨 시스템이나 서비스를 의미함
- 핵심: 단순히 데이터만 주고받는 것이 아니라, 주소와 메서드만 보고도 그 자원이 무엇인지, 어떤 행위를 할 수 있는지 쉽게 알 수 있도록 설계하는 것
- 3대 규칙
  1. 자원의 식별: 모든 데이터(자원)는 고유한 URI(주소)로 식별되어야 함
    - 예: /users/123 (123번 사용자), /products/456 (456번 상품)
  2. 메서드 활용: 조회(GET), 생성(POST), 수정(PUT), 삭제(DELETE) 등의 HTTP 동사 메서드를 적절히 사용
    - 예: GET /users/123 (123번 사용자 조회), POST /products (새 상품 생성)
  3. 자체표현: 서버는 자원의 상태를 클라이언트가 이해할 수 있는 형식(주로 JSON)으로 표현하여 전달
    - 예: {"id":123, "name":"John Doe", "email":""}
- 나쁜 예시
  1. GET /users?id=1: 동사가 주소에 포함되어있음 
  2. POST /updateProduct?id=1: 행위가 주소에 섞여있어 복잡함
- 좋은 예시
  1. GET /products/1: 주소로는 명사로만, 행위는 메서드로 구분
  2. PUT /products/1: 1번 상품을 수정하겠다는 의도가 주소만 봐도 명확

## HTML(HyperText Markup Language)
- HyperText(초월적인 텍스트): 단순한 글자가 아닌 클릭하면 **다른 곳으로 이동할 수 있는 '링크' 기능을 가진 텍스트**
- Markup Language(마크업 언어): 데이터를 설명하는 태그를 추가하는 언어
- 예: <a href="https://www.example.com">Example</a>

## 렌더링
- 뜻: 텍스트로 된 코드(HTML, CSS, JavaScript 등)를 브라우저가 해석하여 시각적으로 보여주는 과정

## SSR(Server-Side Rendering)
- server-side(서버 측): 서버에서 처리하는 것 **서버가 주도권을 가짐**
- 뜻: 서버 측 렌더링은 사용자가 웹페이지에 접속했을때, 서버에서 해당페이지를 완전히 렌더링하고 완성된 HTML 문서로 클라이언트에게 제공하는 기술
- 예시: 전통적인 웹사이트, 블로그, 뉴스 사이트 등
- 특징: 빠른 초기 로딩 속도, SEO에 유리, 서버 부하 증가

## CSR(Client-Side Rendering)
- client-side(클라이언트 측): 사용자의 브라우저에서 처리하는 것 **클라이언트가 주도권을 가짐**
- 뜻: 서버는 텅 빈 HTML과 데이터(JSON)만 보내주고, 클라이언트(브라우저)가 자바스크립트를 이용해 페이지를 동적으로
  렌더링하는 기술
- 동작 과정:
  1. 요청: 사용자가 페이지에 접속
  2. 응답: 서버는 내용이 거의 없는 뼈대 HTML과 필요한 데이터(JSON)를 보냄
  3. JavaScript 실행: 브라우저가 자바스크립트를 실행하여 데이터를 받아와서 페이지를 완성
  4. 렌더링: 브라우저가 받아온 데이터를 바탕으로 화면의 내용을 채워넣음

## PK와 FK
 1. PK (Primary Key, 기본키) / 쉬운뜻: 신분증
테이블에서 각 행(Row)을 유일하게 식별할 수 있는 핵심 컬럼.
중복 불가 (Unique): 사물함 번호나 주민등록번호처럼 똑같은 값이 두 개 존재할 수 없음.
비어있음 불가 (Not Null): 값이 반드시 있어야 하며, 빈칸(NULL)으로 둘 수 없음.
예시: 회원 테이블에서 'ID' 컬럼이 PK 역할을 함. 각 회원은 고유한 ID를 가져야 하며, 중복되거나 비어있을 수 없음.

2. FK (Foreign Key, 외래키) / 쉬운뜻: 권한 설정
다른 테이블의 PK를 참조하는 컬럼입니다. 테이블과 테이블 사이의 연결 다리 역할을 함.
관계 형성: 예를 들어 '게시판' 테이블에 '작성자ID'라는 컬럼이 있다면, 이 값은 반드시 '회원' 테이블의 PK인 'ID'에 존재하는 값이어야 합니다.
데이터 무결성: 회원가입도 안 한 사람이 게시글을 쓸 수 없도록 방지하는 장치입니다.
참조 무결성: 만약 FK로 연결되어 있다면, 게시글이 남아있는 회원을 함부로 삭제할 수 없게 막아주기도 합니다.
예시: '게시판' 테이블의 '작성자ID' 컬럼이 '회원' 테이블의 'ID' 컬럼을 참조하는 FK 역할을 함.

3. 요약 : PK가 존재해야만 FK가 의미가 있음. PK는 각 행을 고유하게 식별하는 역할, FK는 다른 테이블과의 관계를 형성하는 역할을 함.

## JAVA의 5대 핵심요소
1. SOLID 원칙
- SOLID는 다섯가지 원칙의 앞글자를 딴 약어.
S: Single Responsibility Principle(단일 책임 원칙)
 - 클래스 한개는 하나의 책임만 가져야 한다.
O: Open/Closed Principle(개방-폐쇄 원칙)
 - 소프트웨어 개체(클래스, 모듈 등) 는 확장에는 열리고, 수정때는 닫혀야 한다.
L: Liskov Subsitution Principle(리스코프 치환 원칙)
 - 자식 클래스는 언제나 자신의 부모 클래스로 교체할 수 있어야 한다.
I: Interface Segregation Principle(인터페이스 분리 원칙)
 - 특정 클라이언트를 위한 인터페이스
D: Dependency Inversion Principle(의존성 역전 원칙)
- 고수준이 저수준을 찾아서는 안되며, 중간에 있는 약속(인터페이스)에 의존해야 한다.
  ex: 자동차가 타이이어 회사에 직접 의존하는 것이 아니라, 타이어 회사가 제공하는 '타이어'라는 약속(인터페이스)에 의존하는 것
